1. Data-Access-Framework

OFBiz verwendet ein eigenes ORM-ähnliches Framework statt JPA/Hibernate. Der Zugriff erfolgt ausschließlich über drei zentrale Abstraktionen:

// Lesezugriff — Fluent Query Builder
GenericValue order = EntityQuery.use(delegator)
    .from("OrderHeader")
    .where("orderId", orderId)
    .cache()   // aus Entity-Cache
    .queryOne();

// Schreibzugriff — GenericDelegator
delegator.create(newValue);    // INSERT
delegator.store(changedValue);  // UPDATE
delegator.storeAll(valueList); // Batch INSERT/UPDATE
delegator.remove(value);      // DELETE
// Streaming großer Ergebnismengen
try (EntityListIterator it = EntityQuery.use(delegator)
        .from("Product").where(cond).queryIterator()) {
    GenericValue v;
    while ((v = it.next()) != null) { … }
}

4. Transaktionsverhalten

Transaktionssteuerung
Explizite Transaktionen TransactionUtil.begin() / .commit() / .rollback() werden in 96 Stellen explizit aufgerufen — überwiegend in komplexen Multi-Entity-Operationen (z. B. Auftragsanlage, Zahlungsabgleich).
Deklarative Transaktionen Jeder Service in OFBiz trägt in seiner XML-Definition die Attribute useTransaction="true" und optional requireNewTransaction="true". Der ServiceDispatcher eröffnet und schließt die Transaktion automatisch.
Rollback-Strategie Bei GenericEntityException wird in 112 Stellen explizit rollback() aufgerufen. Der Dispatcher rollt bei einem ServiceUtil.returnError() die Transaktion ebenfalls automatisch zurück.
Nested Transactions TransactionUtil.suspend() / .resume() ermöglichen das temporäre Aussetzen der laufenden Transaktion für unabhängige Sub-Operationen (z. B. Audit-Log-Einträge, die bei Rollback erhalten bleiben sollen).
Timeout transactionTimeout pro Service konfigurierbar (Sekunden). Standard: kein globales Timeout gesetzt — Risiko bei lang laufenden Batch-Services.
Typisches Transaktionsmuster
// Explizites Muster (Entity-Schicht)
boolean began = TransactionUtil.begin();
try {
  delegator.create(a);
  delegator.storeAll(list);
  TransactionUtil.commit(began);
} catch (GenericEntityException e) {
  TransactionUtil.rollback(began, msg, e);
  return ServiceUtil.returnError(…);
}

// Deklaratives Muster (Service-Schicht)
// <service useTransaction="true">
// → ServiceDispatcher verwaltet begin/commit

5. Fehlerbehandlung

Rückgabe-Typen
MethoderesponseMessageAufrufe
returnSuccess() success 598
returnError(…) error 2608
returnFailure(…) fail 48

Das Verhältnis 2608 Fehler : 598 Erfolg (ca. 80 % Fehlerpfade) zeigt eine sehr defensive Programmierung — nahezu jeder Aufrufpfad hat explizite Fehlerbehandlung.

Exception-Hierarchie
ExceptionSchichtBedeutung
GenericEntityException Entity Alle Datenbankfehler (SQL, Verbindung, Constraint)
GenericServiceException Service Service-Aufruffehler, falsche Parameter, Engine-Fehler
GeneralException Base Basisklasse beider obiger Exceptions
Map returnError() Service Kein throw — Fehler als Map-Return propagiert
Anti-Pattern: Viele Services fangen Exception e statt spezifischer Typen und geben ServiceUtil.returnError(e.toString()) zurück. Stack-Traces werden so verschluckt — erschwertes Debugging in Produktion.

6. Caching-Strategie

Cache-Typ In-Process Entity-Cache (Caffeine-basiert). Kein verteilter Cache.
Aktivierung .cache() auf EntityQuery-Instanz — 505 Stellen im Code.
Invalidierung Automatisch bei create(), store(), remove() über den Delegator.
Gecachte Entities Konfigurationsdaten, Status-Typen, Referenzlisten — nicht Transaktionsdaten wie Orders.
Cluster-Risiko Bei Multi-Node-Betrieb kein Cache-Sync zwischen Knoten. Tech Debt
// Gecachte Konfigurationsabfrage
GenericValue store = EntityQuery.use(delegator)
  .from("ProductStore")
  .where("productStoreId", storeId)
  .cache()
  .queryOne();

// Nicht gecacht — Transaktionsdaten
GenericValue order = EntityQuery.use(delegator)
  .from("OrderHeader")
  .where("orderId", orderId)
  .queryOne(); // kein .cache()
